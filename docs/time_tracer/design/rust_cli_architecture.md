# Architecutre Decision: Rust as the CLI Presentation Layer

## 1. 为什么选择 Rust 替代 C++ 构建 CLI 表现层？

在 `time_tracer` 的架构中，核心业务逻辑被封装在 `time_tracer_core.dll` 内部，统一对外暴露 C ABI（如 `tracer_core_runtime_ingest_json` 等）。
在这种“核心引擎 + 多端表现层”的 Clean Architecture 下，采用 Rust 来重写原先由 C++（CLI11）编写的命令行客户端（CLI），具有显著的架构优势。Rust 的语言特性非常适合用于编写系统级的客户端表现层：

### 1.1 卓越的安全 FFI（外部接口）机制
由于核心功能由 C++ DLL 提供，CLI 表现层的首要任务是**安全地调用 C ABI**。
*   **C++ 的局限**：在 C++ 中调用 C 接口，需要手动管理生命周期、处理原始指针（raw pointers）和字符串缓冲区，对内存安全的管理要求较高。
*   **Rust 的优势**：Rust 的 `libloading` 和安全的 FFI 封装能力，可以在加载 `.dll` 后，利用 Rust 的所有权和生命周期（Lifetimes）机制，将 `const char*` 边界转换为安全的 Rust `String` 或引用。这在系统的表现层边界建立了一层强有力的内存安全保障。

### 1.2 现代化的错误处理模型
CLI 工具的用户体验很大程度上依赖于错误反馈的清晰度。
*   **C++ 的局限**：异常（Exceptions）处理机制相对隐式，未捕获的异常可能导致程序直接终止并输出堆栈信息，对终端用户不够友好。
*   **Rust 的优势**：通过 `Result<T, E>` 和强类型的模式匹配（Pattern Matching），Rust 要求开发者在编译期显式处理失败场景。配合 `?` 运算符，错误可以结构化地向上传递，并统一格式化为最终的终端输出提示。

### 1.3 强类型的枚举与模式匹配
命令行应用的核心是**指令路由（Command Routing）**。
*   **C++ 的局限**：实现命令分发通常依赖嵌套的条件判断或基于继承的虚函数多态分发，在扩展新命令时缺乏编译器的穷尽检查提示。
*   **Rust 的优势**：强大的 `enum` 和带有**穷尽性检查（Exhaustiveness Checking）**的 `match` 语句，使得新增 CLI 子命令时如果在 `match` 中遗漏处理分支，将产生编译错误。这从语言级提升了命令路由的正确性。

### 1.4 内置的原生字符串与文本处理
CLI 的核心交互模型涉及大量的文本参数解析与格式化输出。
*   **C++ 的局限**：标准库的字符串分割处理、正则表达式和 UTF-8 支持相对基础，复杂处理往往需要引入较大的第三方包或大量定制代码。
*   **Rust 的优势**：标准库提供了完善且内存安全的字符串切片与 UTF-8 编码支持，能够高效地满足表现层的文本处理需求。

## 2. 为什么选择 `clap` 替代传统的 CLI 解析库？

除了 Rust 语言特性外，`clap` 框架提供了极具生产力的开发模式。

### 2.1 声明式 vs 命令式
*   **传统框架（命令式）**：通常需要手动实例化 App 对象，通过注册回调函数和链式调用绑定变量，配置逻辑随应用规模增长容易呈现发散状态。
*   **clap（声明式/宏生成）**：通过 `#[derive(Parser)]` 过程宏，开发者只需定义清晰的数据结构。`clap` 会自动将其映射为对应的解析流程，实现“数据结构即配置”。

### 2.2 自动生成标准化的帮助文档
*   **传统框架**：对于复杂的子命令系统，维持长选项格式、多级说明、甚至高亮颜色的统一，需要耗费一定的格式化适配成本。
*   **clap**：基于定义好的数据结构和注释文档，能够零配置生成符合 Unix 标准规范、布局规整的手册与指南。

### 2.3 规范化的入参校验前置
*   **传统框架局限**：细粒度的数据校验（如数值越界判断或模式匹配）有时需在业务层后置处理。
*   **clap**：可在属性定义处通过 `value_parser` 等方式挂载验证器，让解析器在前排清洗数据，确保透传到核心接口层的数据合法性。

### 2.4 生态福利：自动补全机制
*   用 `clap` 可以在几行代码内生成 Bash / Zsh / PowerShell / Fish 的自动补全脚本（Autocomplete）。要用 C++ 库实现同等的全平台自动补全支持，工作量是无法想象的。

---

## 3. 最终架构愿景

综上所述，**C++ 负责提供高性能运算和底层业务逻辑沉淀，而 Rust 负责安全管理外围接口、进行严格的输入清洗以及实现清晰流畅的展现逻辑**。

未来演进中，Rust CLI 应**稳步剥离**对遗留进程（如 `time_tracer_cli.exe`）的代理依赖，包括独立接管 `crypto` 及其它全体系命令交互。CLI 将仅通过 `c_abi` 接口与核心引擎直接通信，进而实现系统各层的完全解耦与技术栈责任对齐。
