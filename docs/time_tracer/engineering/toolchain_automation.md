# Time Tracer：工程自动化流水线技术文档

本文档描述 `time_tracer` 项目基于 Python 构建的工程自动化体系，包含编译构建、质量保障与测试验证流程。

---

## 1. 编译构建：Python 驱动的 CMake 编排

项目通过 `tools/run.py` 调度 CMake 构建过程，替代手动命令行操作。

### 核心逻辑 (tools/toolchain/commands/cmd_build/cmake.py)
*   **状态感知 (State-Aware)**：解析 `CMakeCache.txt` 确定配置状态，根据标志位（如 `ENABLE_CLANG_TIDY`）或路径变化自动触发重新配置。
*   **并发编译**：根据系统核心数自动配置并行任务数 (`-j`)。
*   **AI Agent 协作**：Agent 通过单一指令（如 `python tools/run.py verify ...` 或 `python tools/run.py validate ...`）触发配置、编译、产物同步及测试全流程。

---

## 2. Clang-Tidy：批次处理与增量刷新

针对大型项目静态检查耗时问题，实现批次化管理逻辑。

### 自动化批次处理 (tools/toolchain/commands/tidy/batch.py)
*   **分段检查点 (Checkpointing)**：流水线划分为 `verify` -> `clean` -> `refresh` -> `finalize` 阶段，支持在中断后从检查点恢复。
*   **任务分片**：将错误日志分割为若干批次（Batch），每个批次由多个任务文件（Task Log）组成。

### 实时刷新机制 (tools/toolchain/commands/tidy/refresh.py)
*   **增量推理 (Auto-Reasoning)**：通过 Mapper 逻辑识别受源代码变动影响的编译单元。
*   **频率控制 (Cadence)**：通过 `full_every` 机制在多次增量刷新后强制执行全量刷新。

### Agent 维护工作流 (Workflows)
项目为 Agent 定义了标准化的维护工作流，用于处理静态检查任务队列：
*   **全量修复流 (`clang_tidy_all`)**：采用 `tidy-flow` 指令进行流水线作业，目标是清空所有任务队列，直至满足“任务日志全清”的硬结项门禁。
*   **定量/定额修复流 (`clang_tidy_by_num`)**：支持按任务数 (Log mode) 或批次数 (Batch mode) 进行局部修复，适用于小步快跑的维护模式。
*   **核心约束**：代码修改必须遵循 ABI 边界抑制政策（仅限 FFI 边界定点抑制），并强制在每个 Task 或 Batch 修复后执行 `artifact` 级验证。

---

## 3. LOC 阈值监控：代码健康管理

基于 `python -m tools.devtools.loc_scanner` 监控代码分布。

### 监控维度 (scan_lines.toml)
*   **文件行数监控**：按语言设定不同阈值（C++: 350, Python: 250），识别需重构的大文件。
*   **目录文件数监控**：监控单一目录下的文件密度，提示分层细分需求。
*   **精度控制**：通过 `over_inclusive` 开关控制边界判定逻辑。

---

## 4. 测试与验证：覆盖率策略

测试框架采用“二层模型”覆盖核心算法与交付产物。

### 二层测试模型 (docs/toolchain/test/README.md)
1.  **逻辑测试层**：`tracer_core` 的 C++ 单元与组件测试。
2.  **产物测试层**：针对二进制交付件（如 `time_tracer_cli.exe`）的集成测试。

### 命令行命令 100% 覆盖策略 (test/suites/tracer_windows_rust_cli)
*   **契约化断言**：通过 TOML 定义命令行为，支持 JSON 字段路径校验。
*   **结构化反馈**：系统输出 `result.json`，Agent 可通过解析 JSON 状态判定结果。
*   **自动格式修正**：验证通过后自动执行 `clang-format`。
*   **异常环境模拟 (Runtime Guard)**：模拟异常运行环境（如文件缺失），覆盖错误处理路径。

## 5. 基于 LLM 视角的风格一致性分析

从大语言模型（LLM）逻辑分析，物理风格的一致性对生成质量有以下影响：

*   **减少干扰 Token**：统一缩进可减少非语义 Token 对预测概率的影响，提高逻辑生成准确度。
*   **模式匹配强化**：一致的命名规范（如变量后缀、异常宏）有助于 LLM 准确预测代码结构。
*   **维持上下文一致性**：统一风格提供恒定的结构参考，降低长文本处理中的风格偏移。
*   **优化 检索增强生成 (RAG) 融合度**：确保检索的代码片段与当前位置在结构上兼容，降低合并时的语义与语法错误。

## 6. 认知成本与工程可靠性

静态辅助工具通过一致性约束降低开发与维护成本：

*   **对人类开发者**：减少代码评审中的视觉干扰，使关注点集中于逻辑。
    *   **降低操作风险与 Token 消耗**：将复杂的指令链、文件搬运、产物复制等重复性物理操作封装在 Python 脚本中。具体的自动化减负示例包括：
        *   **动态环境治理**：`Context.setup_env()` 自动处理 `JAVA_HOME` 等跨平台环境变量，避免 Agent 因环境缺失导致构建失败。
        *   **配置自动同步**：通过 `config_sync.py` 在构建前自动将核心通用配置分发至 Windows/Android 特定目录，防止 Agent 遗漏物理拷贝环节。
        *   **二进制路径自愈**：测试框架 (`suite.py`) 会根据构建模式自动解析并校验二进制路径（如 `_resolve_effective_bin_dir`），Agent 无需在深层构建目录中手动“人肉”查找产物。
        *   **冷热启动感知**：`cmake.py` 通过解析 `CMakeCache.txt` 自动识别配置偏差并触发重配置，Agent 仅需发出“构建”指令，无需判断是否需要“清理/重新配置”。

## 7. 架构屏障与规模化治理 (Scale & Boundary Enforcement)

对于一个拥有 60,000+ 行 C++ 代码以及跨平台 UI/CLI 的复杂项目，这套流水线是 AI Agent 能深度参与的**物理前提**：

*   **突破“规模门槛” (The Scale Barrier)**：在 6 万行 LOC 级别，若无严格的自动化约束，任何 Agent 都会在极短时间内迷失在上下文碎片中。自动化门禁充当了项目的“外骨骼”，确保 Agent 仅需关注局部逻辑。
*   **物理边界强制生效 (Physical Layer Isolation)**：
    *   **核心逻辑隔离**：Android UI 和 Windows CLI 仅作为表现层。
    *   **单向依赖契约**：通过构建系统强行确保 UI 与 CLI **只能调用 Core**，禁止 Core 反向依赖任何表现层。这种“单向阀”设计极大地降低了 Agent 在修改 Core 时产生连锁副作用的概率。
*   **Agent 的“导航系统”**：如果没有这些高度一致的结构化反馈，即使是最顶级的 Agent，其 80% 的工作量也会消耗在处理“编译器报错”或“路径拼写错误”等低价值细节上。

## 8. 工程化深度与 Agent 协作效能分析 (Efficiency Analysis)

在 6 万行以上规模的 C++ 工程实践中，维持代码质量与演进速度面临显著的复杂性挑战。本项目的自动化体系在处理此类规模时表现出以下物理特征：

*   **隐性经验的显性化与自动化**：本项目通过 Python 脚本将环境配置、产物分发、构建状态检查等工程经验转化为可执行代码。这种显性的自动化机制降低了对操作者依赖“大脑缓存”的要求。
*   **物理层面的质量约束**：随规模增长，协同成本通常会上升。本项目通过 LOC 阈值监控和 Clang-Tidy 批次化处理，在物理架构层面引入了强制性的质量门禁，为指令迭代提供了一个**低干扰、高确定性**的反馈循环。
*   **面向 AI 协作的契约设计**：不同于传统的以文档为中心的维护模式，本项目采用结构化（JSON）的反馈契约与自动化的物理环境治理。这种设计减少了 Agent 获取系统状态的路径，使其能够专注于跨模块重构（如 C++ Modules 迁移）等复杂逻辑任务。

## 9. 总结

`time_tracer` 工程自动化体系不仅是工具集，更是一种**工程契约**：
*   **构建自动化**：路径解析与环境检查由脚本处理，人机体验高度一致。
*   **质量可量化**：通过批次静态检查与 LOC 监控管理技术债。
*   **验证标准化**：通过契约化测试，确保 Android 与 Windows 两大表现层在 60k+ 行代码底座上始终保持稳健。
