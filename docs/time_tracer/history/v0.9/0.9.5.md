## [v0.9.5] - 2026-03-30

### 新增功能 (Added)
* 新增 `libs/tracer_core` 原子记录接口 `RecordActivityAtomically`，用于承接 Android 侧按当前时间生成单行活动记录的录入方式，并统一提供 `target_date_iso`、`raw_activity_name`、`remark`、`preferred_txt_path` 与 `date_check_mode` 入参，以及 `ok`、`message`、`operation_id`、`warnings`、`rollback_failed`、`retained_transaction_root` 出参。
* 新增 `libs/tracer_core/src/domain/types/time_order_mode.hpp` 与配套模块导出，定义 `strict_calendar`、`logical_day_0600` 两种显式时间排序模式，用于把“目标日语义”与“时间比较语义”解耦并显式透传。
* 新增 `libs/tracer_transport/include/tracer/transport/runtime_requests.hpp` 的可选字段 `time_order_mode`，用于把上层 `yesterday/today` 决策无损传递到 libs 记录比较核心，不在桥接层隐式猜测模式。
* 新增 `wake_keywords` 与 `authorable_event_tokens` 对外查询链路（`libs/tracer_core` → `tracer_transport` → C API/JNI），用于把“wake 语义集合”和“作者态可输入 token 集合”从 core 配置统一下发，解决 Android 侧只能按 `alias_mapping.toml` 左键校验、导致 wake 词无法录入的问题。

### 技术改进/重构 (Changed/Refactor)
* 重构 `libs/tracer_core/src/application/pipeline/pipeline_workflow.cpp` 的记录提交流程，用于桥接“Android 单行生成式记录”和“core 按整份 TXT 校验”的语义差异，改为“内存构造候选 TXT -> 临时文件整份校验 -> 备份正式 TXT -> 覆盖正式 TXT -> `single_txt_replace_month` ingest -> 失败回滚”的原子事务模型。
* 重构 `libs/tracer_transport`、`apps/tracer_core_shell/api/c_api` 与 `apps/tracer_core_shell/api/android_jni` 的运行时桥接链路，为原子记录接口补齐 JSON 请求编解码、C ABI 导出和 JNI 调用入口。
* 重构 `libs/tracer_core/src/application/pipeline/detail/pipeline_record_time_order_support.hpp` 的时间比较模型，改为“`HHmm -> 分钟轴映射 -> 严格递增比较`”的通用模式算法，并以命名常量固定 `06:00` 切日点，用于消除 `0009 > 2058` 这类时间对硬编码特判。
* 重构 `libs/tracer_core` 记录实现拆分：将 atomic record helper、alias 校验与文本构建逻辑拆到 `pipeline_record_atomic_support_impl.inc` 与 `pipeline_record_alias_text_support_impl.inc`，使 `PipelineWorkflow` 仅保留 orchestration，用于降低业务耦合并提升可测性。
* 重构 `apps/tracer_core_shell/api/c_api/runtime/c_api_parse_bridge.cpp` 与 JNI 解析链路，对 `time_order_mode` 执行“默认 `strict_calendar` + 非法值显式报错（含允许值）”策略，用于保证兼容旧调用方并避免静默错误落库。
* 重构 `libs/tracer_core/src/infra/query/data/repository/query_runtime_service*.cpp` 的配置查询能力，明确拆分 `mapping_alias_keys`、`wake_keywords`、`authorable_event_tokens` 三种语义，解决过去“普通活动映射集合”和“wake 语义集合”混在一起，导致作者态校验口径不清的问题。
* 重构 `libs/tracer_core/src/application/pipeline/detail/pipeline_record_alias_text_support_impl.inc` 的记录名校验与 completeness warning 逻辑，改为“wake 分类只看 `wake_keywords`，作者态可输入集合才使用并集”，用于消除“wake 识别依赖 alias 映射”这一业务歧义。
* 重构 `libs/tracer_core/src/domain/logic/validator/structure/structure_validator.cpp` 的有效性判定，改为“只有 `kError` 才失败”，用于给作者态 warning 留出正式语义边界，而不是继续沿用“只要有诊断就失败”的旧口径。

### 修复 (Fixed)
* 修复新的一天只能先记录第一条活动、却被 core `activity.count.too_few` 阻断的问题：libs 不再把“当天少于 2 条 authored events”当作 ingest 非法输入，而是仅在作者态保留 completeness warning，解决了单行记录被业务规则反向卡死的死循环。
* 修复 `Record Activity` 失败时可能留下半成功状态的问题：活动名 alias 不匹配、候选 TXT 结构/逻辑校验失败或 ingest 失败时，正式 TXT 与 DB 都保持调用前状态。
* 修复跨午夜时间递增判定错误：在 `logical_day_0600` 下允许 `2058 -> 0009`、`2207 -> 0200` 等活动日内递增序列，在 `strict_calendar` 下保持原自然日比较语义。
* 修复记录输入与文本解析职责混淆问题：移除 `pipeline_workflow.cpp` 中硬编码 alias 文本映射（例如 `"zh" -> "zhihu"`），改为记录阶段仅做 alias 合法性校验，实际名称转换仅在整文解析/入库阶段按 `alias_mapping.toml` 执行。
* 修复 wake 词只能在 `alias_mapping.toml` 中存在时才可被单行记录接受的问题：core atomic record 现已接受“`alias key` 或 `wake keyword`”两类 token，解决了 wake 语义明明由 `interval_processor_config.toml` 定义、作者态却无法直接录入 wake 的业务断裂。
* 修复 `sleep_night` 相关 warning 与 wake 判断口径不一致的问题：record candidate 与整月 completeness 扫描改为只基于 `wake_keywords` 判断“首条是否 wake”，避免本地硬编码或 alias 参与 wake 分类，导致通宵/起床语义判断偏差。
